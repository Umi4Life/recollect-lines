# RFC-004 — Unified Durable Supervision for CLI Runtimes

**Status:** Proposed

**Authors:** Recollect Lines maintainers

**Date:** 2026-07-22
**Decision:** Every subprocess-backed CLI runtime MUST run behind one shared, crash-durable subprocess supervisor. Runtime adapters provide a minimal launch specification and runtime-specific result parser only.

## 1. Problem statement

Recollect Lines currently has two process-lifecycle designs:

1. the production `legacy_subprocess` path, used by Cursor, Claude Code, Codex, and OpenCode; and
2. a more durable supervisor/reconciliation design that is presently wired only through a fixture adapter.

The production path gives each adapter a direct in-memory `subprocess.Popen` handle. Completion, reaping, artifact refresh, restart recovery, and cleanup are therefore distributed across adapter code, `Broker`, completion-event polling, `collect()`, and reconciliation. The broker cannot reliably answer whether a task is still running, has exited, or has produced recoverable work.

This RFC replaces the split design with one authoritative lifecycle model for all CLI adapters.

### 1.1 Observed failures

The following findings were reproduced or directly established from source at `ce153d3`.

| Incident | Observable symptom | Confirmed cause |
|---|---|---|
| Same broker instance, exited child | Process becomes zombie; `status()` and `reconcile()` keep returning `running` | Legacy handles are only reaped by `collect()`/selected completion-event paths. `status()` does not poll. `reconcile()` returns early when an in-memory handle exists. |
| macOS broker restart | A live Cursor leader is classified dead; reconciliation may release its worktree | The Darwin process "identity" includes `time.monotonic_ns()` sampled at each read. Two reads of one unchanged live PID cannot match. |
| Broker restart after real completion | Task becomes `uncollected`; result cannot be recovered; evidence/worktree can be discarded | Legacy `Popen` handle and terminal result are not durable. Restart loses the only normal collection path. |
| Artifact reporting | `status()` reports zero-byte output despite real runtime output on disk | `manifest.json` is cached at launch and refreshed during terminal collection, but not by ordinary status/reconcile reads. |
| Premature collection | One `collect()` call can stall all MCP callers | The MCP server is synchronous/single-threaded and adapter `collect()` calls unbounded `Popen.wait()`. |

These are not Cursor-only defects. Cursor made them visible because its tasks are long-lived, but the underlying lifecycle contract is shared by all current CLI adapters.

## 2. Goals

1. Every CLI task has a durable, restart-safe lifecycle record before `delegate` reports that it started.
2. Every task reaches exactly one durable terminal lifecycle state: `exited`, `cancelled`, `timed_out`, or `lost`/`unknown`.
3. A normal read API (`status`, completion events, or collection) can never leave an already-exited child unreaped merely because a caller chose the wrong polling endpoint.
4. A broker restart can recover exit status, output artifacts, and process liveness without depending on a former process's Python object.
5. Runtime adapters become thin integrations: construct a command and parse their own result format.
6. No liveness uncertainty can directly delete a worktree or workspace.
7. `collect()` becomes non-blocking for a live task and cannot head-of-line block the MCP server.
8. The design supports future CLI providers without copying process-management code.

## 3. Non-goals

- Replacing provider-specific command-line flags, authentication, or structured output parsers.
- Promising a portable stable process creation identity on every operating system before it has been implemented and tested.
- Automatically deleting failed/uncollected workspaces.
- Changing product-level task semantics, council routing, or provider capability policy.

## 4. Decision

### 4.1 Single durable supervisor

`DurableSubprocessSupervisor` becomes the sole authority for CLI process lifecycle:

```text
persist launch intent
→ create artifacts and durable launch record
→ spawn supervised leader/process group
→ persist PID and validated identity where available
→ stream/capture stdout and stderr to durable artifacts
→ detect exit and reap leader
→ persist exit result and terminal lifecycle event
→ permit restart-time adoption/reconciliation
→ await explicit workspace-release authorization
```

The supervisor—not a runtime adapter and not an MCP request handler—owns:

- process spawning and process-group control;
- output path creation and artifact freshness;
- exit detection, `waitpid`/reaping, exit-code persistence;
- timeouts and cancellation escalation;
- restart-time discovery and recovery;
- liveness classification;
- terminal lifecycle events.

### 4.2 Minimal adapter contract

Every CLI adapter implements only two required capabilities:

```python
class CliRuntimeAdapter(Protocol):
    name: str

    def build_launch_spec(self, request: RuntimeLaunchRequest) -> LaunchSpec:
        """Return argv, cwd, environment allowlist, output contract, and policy metadata."""

    def parse_result(self, completed: CompletedLaunch) -> RuntimeResult:
        """Interpret artifacts/exit code using this runtime's documented output format."""
```

`LaunchSpec` is declarative. It must contain no process object, thread, callback, or background task:

```python
@dataclass(frozen=True)
class LaunchSpec:
    argv: tuple[str, ...]
    cwd: Path
    environment: Mapping[str, str]
    stdin_mode: Literal["closed", "input_file"] = "closed"
    stdout_name: str = "stdout.log"
    stderr_name: str = "stderr.log"
    result_contract: ResultContract = ResultContract.TEXT
    grace_period_seconds: float = 10.0
    timeout_seconds: float | None = None
    process_group: bool = True
```

Adapters MAY provide runtime diagnostics and result parsing helpers. They MUST NOT call `Popen`, `wait`, `poll`, `killpg`, write lifecycle manifests, infer liveness, or release workspaces.

### 4.3 Runtime migration target

| Runtime | Adapter retains | Supervisor assumes |
|---|---|---|
| Cursor | `cursor-agent` argv, sandbox/worktree flags, Cursor JSON-lines parser | launch, output capture, exit/reap, restart recovery |
| Claude Code | `claude` argv and output parser | same |
| Codex | `codex` argv and output parser | same |
| OpenCode | `opencode` argv and events/result parser | same |
| Future CLI provider | minimal `LaunchSpec` builder and parser | same, with no new lifecycle implementation |

## 5. Lifecycle and public semantics

### 5.1 State model

The durable launch state is distinct from product outcome:

```text
launching → running → exited_uncollected → collected
                    ↘ cancelled / timed_out
                    ↘ recovery_required / lost
```

Rules:

- `running` means a supervisor has verified the leader remains running, not merely that a stale record says so.
- `exited_uncollected` means the process was reaped and artifacts are durably available; no success claim has been made.
- `recovery_required` means liveness or result recovery is uncertain. It is never treated as dead for cleanup.
- `lost` is reserved for evidence-backed irrecoverability, not PID/identity guesswork.
- `collect()` on `running` returns immediately with `not_ready`; it never waits for process completion.

### 5.2 Liveness policy

Liveness is tri-state/four-state, never a Boolean inferred from a PID string:

```text
running
exited
missing_or_reused
unknown
```

Required policy:

- Parent-owned children are checked with a nonblocking reap primitive where supported (`waitpid(..., WNOHANG)` on POSIX).
- A zombie is `exited`, not `running`.
- Linux may use `/proc` boot ID plus process starttime as a stable supplemental identity.
- Until a correct Darwin process-start identity is implemented and tested, Darwin identity mismatches return `unknown`, never `dead`.
- `unknown` transitions to `recovery_required` and preserves the workspace.

### 5.3 Cleanup policy

`reconcile()` is observational/recovery-oriented. It MUST NOT delete a worktree.

Workspace release becomes a separate explicit operation:

```text
release_workspace(task_id, confirmation / retention-policy eligibility)
```

It requires a terminal or operator-confirmed recovery state plus a final liveness check. A grace period is recommended for `uncollected` tasks so an operator can inspect or salvage the artifact/worktree.

### 5.4 Artifacts and results

- Output paths are persisted before the child starts.
- Artifact metadata either refreshes on `status()` or reports an explicit `generated_at`/`stale` indicator; it must never imply a live byte count when it is a cached snapshot.
- `collect()` parses durable artifacts and persisted exit status, allowing a new broker instance to recover a result after restart.
- `diff.patch`, workspace status, stdout/stderr, and normalized result are retained under explicit retention policy.

## 6. Compatibility and migration

### 6.1 Feature gate

Introduce an internal launch-mode gate:

```text
cli_supervisor_mode = legacy | durable
```

Default remains `legacy` until a runtime has passed its migration acceptance suite. Production rollouts move one runtime at a time, then switch global default only after all four are durable.

### 6.2 No silent fallback

A runtime configured for `durable` that cannot create a durable launch record fails before child start. It MUST NOT silently fall back to a direct `Popen` path.

### 6.3 Legacy decommissioning

After Cursor, Claude Code, Codex, and OpenCode all use the supervisor:

- remove `legacy_subprocess` launch kind;
- remove adapter `ProcessHandle` ownership;
- remove adapter-specific reconciliation branches;
- delete duplicate process-group and direct-collection code;
- retain any compatibility reader needed only to recover existing legacy tasks during a documented sunset window.

## 7. Implementation plan and task breakdown

### Phase 0 — Containment and regression lock

**Goal:** stop destructive outcomes before the main refactor.

- [ ] **P0.1: Split cleanup from reconciliation**
  - Remove automatic workspace release from every `reconcile()` path.
  - Add an explicit workspace-release API with confirmation/retention checks.
  - Acceptance: a false liveness verdict cannot remove a live or unknown workspace.

- [ ] **P0.2: Correct unsafe liveness classification**
  - Disable the non-Linux monotonic fallback as a proof of death.
  - Classify unsupported-platform identity mismatch as `unknown`.
  - Add zombie-aware parent-owned process detection.
  - Acceptance: a mocked/live Darwin PID sampled twice is not `dead`; a real zombie is `exited`.

- [ ] **P0.3: Make legacy reads safe until migration completes**
  - On `status()` and `reconcile()`, nonblocking-poll known handles and transition/reap exited children.
  - Make `collect()` return `not_ready` for a live process rather than calling unbounded `wait()`.
  - Acceptance: an exited local child observed only through `status()` becomes terminal/reaped; no MCP call blocks on a live child.

- [ ] **P0.4: Artifact freshness contract**
  - Refresh manifest before status or expose manifest generation time/staleness clearly.
  - Acceptance: status byte count matches file bytes after a child writes output, or response explicitly marks cache age.

### Phase 1 — Supervisor contract

**Goal:** make one reusable lifecycle engine production-ready.

- [ ] **P1.1: Define `LaunchSpec`, `CompletedLaunch`, and adapter protocol**
  - Introduce the minimal pure-data launch request/spec contract.
  - Document environment allowlist, stdin, process-group, output, timeout, and result-contract fields.
  - Acceptance: a fixture adapter needs no `Popen` imports or lifecycle callback.

- [ ] **P1.2: Promote durable runner to `DurableSubprocessSupervisor`**
  - Make durable state format/version explicit.
  - Persist launch intent before spawn and exit result before event emission.
  - Add supervisor-owned nonblocking observation and collection APIs.
  - Acceptance: a new broker instance can recover a completed result from the launch record/artifacts alone.

- [ ] **P1.3: Build a supervisor conformance suite**
  - Test normal exit, nonzero exit, timeout, cancellation, orphan/restart recovery, zombie reaping, PID reuse/unknown identity, and output metadata.
  - Run on Linux CI; add macOS CI or an equivalent Darwin contract test before enabling macOS cleanup.
  - Acceptance: every runtime adapter runs the same lifecycle conformance tests.

### Phase 2 — Migrate adapters one by one

**Goal:** eliminate real uses of `legacy_subprocess` without a big-bang rollout.

- [ ] **P2.1: Cursor durable migration**
  - Move argv/worktree/sandbox construction into `CursorAdapter.build_launch_spec()`.
  - Move JSON-lines parsing into `CursorAdapter.parse_result()`.
  - Remove Cursor direct `Popen.wait()` and legacy Cursor reconciliation once parity is proven.
  - Acceptance: broker restart during a real/fixture Cursor task produces `running`, `exited_uncollected`, or `recovery_required` correctly; no worktree is auto-deleted.

- [ ] **P2.2: Claude Code durable migration**
  - Implement spec builder and structured-output parser adapter.
  - Acceptance: same supervisor conformance suite plus Claude-specific output parsing tests.

- [ ] **P2.3: Codex durable migration**
  - Implement spec builder and result parser adapter.
  - Acceptance: same supervisor conformance suite plus Codex-specific output parsing tests.

- [ ] **P2.4: OpenCode durable migration**
  - Implement spec builder and events/result parser adapter.
  - Acceptance: same supervisor conformance suite plus OpenCode event-artifact tests.

### Phase 3 — API, rollout, and deletion

**Goal:** make the durable contract the only supported execution path.

- [ ] **P3.1: Update MCP/task API semantics**
  - Document nonblocking `collect`, recovery states, artifact freshness, and explicit cleanup.
  - Add actionable warnings for `recovery_required` and `exited_uncollected`.
  - Acceptance: callers can distinguish “still running,” “exited but not parsed,” and “unknown” without inspecting raw internals.

- [ ] **P3.2: Controlled rollout**
  - Enable durable mode for Cursor first behind configuration.
  - Dogfood restart/recovery and cancellation scenarios using disposable repos.
  - Enable remaining adapters only after evidence is captured.
  - Acceptance: no automatic downgrade to legacy; telemetry shows terminal-event latency and recovery outcomes.

- [ ] **P3.3: Remove legacy lifecycle**
  - Delete legacy adapter handles, per-adapter lifecycle code, and direct process collection.
  - Remove legacy launch mode after a migration/recovery sunset period.
  - Acceptance: all real CLI runtime starts return a non-null durable launch ID.

## 8. Acceptance scenarios

1. **Status-only completion:** Start a short-lived CLI task; call only `status()`. The task becomes `exited_uncollected` or `collected`, no zombie remains, and output metadata is current.
2. **Restart while running:** Restart broker during a long-running task. The new broker reports `running` only with evidence; otherwise it reports `recovery_required` and leaves workspace intact.
3. **Restart after exit:** Process exits before broker restart. New broker recovers exit code and artifacts, then allows collection without original `Popen`.
4. **Darwin live leader:** A live leader is never marked dead merely because time advanced between two identity reads.
5. **Darwin/unsupported identity:** Unsupported identity produces `unknown` and preserves workspace.
6. **Zombie:** An exited unreaped child is recognized as exited and is reaped by supervisor observation.
7. **Premature collect:** Calling `collect()` on a running task returns promptly and does not block unrelated MCP calls.
8. **Worktree safety:** Reconciliation alone cannot delete a worktree; release requires explicit eligible terminal state and final liveness check.
9. **Future adapter:** A small fake provider can integrate using only `build_launch_spec()` plus `parse_result()` and pass shared conformance tests.

## 9. Rejected alternatives

### Keep legacy `Popen` but call `poll()` from more endpoints

This is needed as Phase 0 containment but is not the final design. It still loses ownership and result state on restart, preserves duplicated lifecycle logic, and makes every new adapter a source of recurrence.

### Let adapters implement a completion callback

Callbacks recreate the same failure class: callback ownership, scheduling, exception handling, and restart semantics differ per runtime. Lifecycle must have one owner.

### Use `kill(pid, 0)` as liveness proof

It cannot distinguish a zombie, PID reuse, or ownership. It is insufficient for destructive decisions.

### Release workspaces after a timeout automatically

A timeout can be a retention policy, but it must follow a terminal/recovery decision and a final conservative liveness check—not be coupled to uncertain reconciliation.

## 10. Operational telemetry

The supervisor SHOULD emit privacy-safe metrics/events for:

- launch persistence before spawn;
- process exit detection latency;
- reaping latency;
- result recovery after restart;
- transition counts by terminal/recovery state;
- workspace release reason and confirmation source;
- artifact-manifest refresh age;
- unsupported identity/unknown-liveness occurrences.

These metrics are diagnostic signals, not substitutes for the durable task record.

## 11. Open questions

1. What stable process creation identity is available through Darwin APIs, and can it be captured without elevated permissions?
2. Should the supervisor run as a detached child process, service-owned worker, or both behind one persisted protocol?
3. What retention period should apply to `exited_uncollected` workspaces by default?
4. Should result parsing run inside the supervisor or as a broker-side deterministic step after durable exit capture? This RFC requires the raw exit/artifact record to be durable either way.
5. Which macOS CI environment can exercise real process/zombie/restart behavior before enabling destructive cleanup there?

## 12. Relationship to earlier design

This RFC does not replace runtime routing, capability discovery, model-provider configuration, or bounded-council behavior. It narrows the execution boundary beneath them: any selected CLI runtime must satisfy one durable subprocess supervision contract.
