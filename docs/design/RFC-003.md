# RFC-003 — Resource and Rework Governance

Status: **accepted decision RFC** (design contract embodied by PRs #55 and #56).
This document records the product and implementation decisions for operator-declared
resource metadata, durable task snapshots, and opt-in bounded rework/escalation
policy. It is **not** an implementation PR.

Related: [PRD.md](PRD.md) (bounded execution, §3.1; parent-directed graphs, §5.6),
[RFC-001.md](RFC-001.md) (broker policy boundary, discovery/council bounds, §10.5),
[RFC-002.md](RFC-002.md) (completion integrity and capability contracts — orthogonal
to cost/rework). Operator guides:
[model-profile-resource-metadata.md](../model-profile-resource-metadata.md),
[cost-rework-policy.md](../cost-rework-policy.md).

## 1. Problem statement and anti-pattern

Parent-directed task trees make it easy to construct workflows that **look**
cheap or local while spending scarce quota, GPU time, or premium model budget.
Without explicit governance, operators and orchestrators fall into predictable
failure modes:

| Anti-pattern | Why it fails |
|---|---|
| **Infer cost from graph position** | A child under a `read_only` parent is treated as low-cost even when the child selects a premium profile or a local model that saturates GPU. |
| **Infer cost from provider/model folklore** | `"gpt-4"`, `"sonnet"`, or `"local"` strings become implicit price tags; renames, gateways, and zero-dollar self-hosted models break accounting. |
| **Live billing lookup inside the broker** | Provider APIs drift, credentials multiply, and preflight becomes non-deterministic — spend may occur before rejection. |
| **Autonomous retries and debate loops** | The broker re-runs or escalates tasks because prose "sounds like" a retry, duplicating satisfied work at higher cost. |
| **Semantic duplicate detection** | Two prompts that mean the same thing are indistinguishable from genuinely new work without NLP the broker does not own. |
| **Review-by-profile** | Selecting `architecture-reviewer` or `review-findings` is mistaken for an automatic review task graph edge or reviewer assignment. |
| **Raw artifact escalation** | Parents inject prior transcripts or files into delegate context instead of declaring rework lineage — bypassing caps and audit. |

**Resource and rework governance** means: cost and scarcity are **operator-declared
metadata** snapshotted per task; rework and escalation are **explicit, opt-in,
budgeted, and preflight-gated** before adapter launch; independent workflow
dimensions are never collapsed into one another.

## 2. Terminology

### 2.1 Independent dimensions (never infer across)

These dimensions are **orthogonal**. The broker must not derive one from another.

| Dimension | What it is | Where it lives | Not inferred from |
|---|---|---|---|
| **Graph role** | Placement in a parent-directed tree: `parent_task_id`, `relationship` (`delegates` / `continues`), `delegation_depth`, `root_task_id`, `external_root_id`, `origin_kind`. | Task lineage ([`task_lineage.py`](../../src/recollect_lines/task_lineage.py)) | Cost class, model profile, agent profile, result schema |
| **Cost class** | Operator-declared coarse spend tier: `low`, `standard`, `premium`, `unknown`. | `model_profiles.*.cost_class`, snapshotted on task | Graph depth, provider name, runtime id, task prose |
| **Resource characteristics** | Five declared scarcity/latency tiers per profile (below). | `model_profiles.*.resources`, snapshotted on task | Monetary folklore, graph role, review role |
| **Capability / authority** | What the runtime may do: `execution_mode`, `tool_access_profile`, `required_capabilities`, runtime capability contract, verification policy. | Task request + runtime registry ([RFC-002](RFC-002.md), [`capability_contract.py`](../../src/recollect_lines/capability_contract.py)) | Cost class, agent profile name |
| **Review role** | Behavioral and output shape for critique: `agent_profile` (e.g. `architecture-reviewer`), `result_schema` (e.g. `review-findings`, `verified-investigation-report`). | Task request + [`agent_profiles.py`](../../src/recollect_lines/agent_profiles.py) | Graph role, cost class, automatic review scheduling |

**Premium budget** always charges the **child task's** resolved `cost_class`, never
the parent's (see `test_graph_role_does_not_inherit_parent_classification`).

### 2.2 Model profile and resource metadata

| Term | Meaning |
|---|---|
| **`model_profile`** | Operator-configured id binding `runtime` / optional `provider` / optional `model` pin to `cost_class`, `usage_bucket`, and `resources`. |
| **`usage_bucket`** | Operator identifier for quota grouping (not a broker-computed bill). |
| **Unconfigured profile** | Omitting `model_profile` → `cost_class: unknown` and all resource tiers `unknown` — not a silent default to `low`. |
| **Snapshot** | Durable `model_profile_resolution.json` written at launch; public projection `model_profile_resource` on status/collect surfaces. |

### 2.3 Resource dimensions

Each profile declares all five tiers explicitly:

| Dimension | Meaning | Example high-cost case |
|---|---|---|
| `monetary_cost` | Operator judgment of direct spend | Cloud API at list price |
| `quota_scarcity` | Rate limits, seat caps, org quotas | Shared org TPM |
| `latency` | Expected wall-clock / tail latency | Large reasoning models |
| `local_compute_occupancy` | GPU/CPU/RAM pressure on operator machine | Zero-dollar local 70B on one GPU |
| `context_cost` | Prompt/context amplification (tokens, retrieval) | Long-repo investigation |

A **zero-monetary-cost** local model may still be `premium` on
`local_compute_occupancy` and `quota_scarcity`.

### 2.4 Bounded rework and escalation

| Term | Meaning |
|---|---|
| **`cost_rework_policy`** | Named operator policy id; opt-in per task. Omitting preserves legacy behavior. |
| **`rework_prior_task_id`** | Prior task in the **same** `root_task_id` workflow. |
| **`rework_scope`** | `targeted` (continuation on prior output) or `full` (re-execution / escalation). |
| **`escalation_reason`** | Bounded operator rationale; presence surfaced on public projections; full text not echoed on all surfaces. |
| **Escalation** | `rework_scope: full` — counts toward `max_escalations_per_workflow`. |
| **Premium retry** | Premium `cost_class` task with rework metadata referencing a prior task — counts toward `max_premium_retries_per_task`. |
| **Satisfied prior** | Prior task in terminal success with satisfied result contract and no failed broker verification — gates higher-cost full duplication. |

### 2.5 Bounded reviewer contracts

Structured result schemas such as `review-findings` and
`verified-investigation-report` ([verified-investigation-report.md](../verified-investigation-report.md))
are **review-shaped output contracts**. They complement human or parent review
but **do not** automatically create review tasks, council rounds, or graph edges.
Raw artifacts remain **escalation-only** — parents fetch via `collect`; the
broker does not inject them into sibling tasks.

## 3. Data and decision flow

### 3.1 Model profile resolution

```text
Operator config (model_profiles)
        │
        ▼
Create/delegate: model_profile id (optional)
        │ validate known id at create
        ▼
Start preflight: runtime/provider/effective_model binding check
        │ reject before adapter launch if incompatible
        ▼
Snapshot → model_profile_resolution.json
        │
        ├─► broker_observed.model_profile_resource (normalized result)
        ├─► concise_normalized_view / completion events
        └─► workflow usage accounting (cost_class per started task)
```

The broker **never**:

- calls provider billing APIs;
- maps model name strings to price;
- upgrades `unknown` → `low` because the runtime is `mock` or local.

### 3.2 Cost/rework preflight (opt-in)

```text
Create/delegate: cost_rework_policy + optional rework metadata
        │ validate policy id + rework field pairing at create
        ▼
Start preflight (only when policy selected):
  1. model_profile must be configured (not unknown)
  2. escalation_reason if policy requires
  3. rework_prior_task_id ∈ same root_task_id
  4. full rework + satisfied prior + higher cost_class → deny unless allow_higher_cost_reexecution
  5. aggregate workflow usage from persisted snapshots (exclude current task)
  6. reject if premium / retry / escalation caps exceeded
        │ reject → TaskState.REJECTED, no launch record, no quota spent
        ▼
Snapshot → cost_rework_policy_resolution.json
        │
        └─► cost_policy_status on status / normalized / completion surfaces
```

Preflight reads **persisted lineage and artifacts**, not task prose semantics.
The broker cannot detect whether a prompt is a semantic duplicate of prior work;
it only enforces caller-declared `rework_scope` and stored `cost_class` history.

### 3.3 Interaction with RFC-002 capability gates

Capability preflight ([RFC-002](RFC-002.md)) and cost/rework preflight are
**separate gates** at start. A task may pass capability checks and fail cost
policy, or vice versa. Capability warnings at collect time do not retroactively
alter cost snapshots or rework budgets.

## 4. Configuration examples

Full field reference: [model-profile-resource-metadata.md](../model-profile-resource-metadata.md),
[cost-rework-policy.md](../cost-rework-policy.md). Illustrative combined operator
config:

```yaml
model_profiles:
  dev-mock-low:
    runtime: mock
    cost_class: low
    usage_bucket: dev-sandbox
    resources:
      monetary_cost: negligible
      quota_scarcity: none
      latency: low
      local_compute_occupancy: low
      context_cost: low

  local-heavy-premium:
    runtime: openai_compatible
    provider: local_gateway
    model: local-70b
    cost_class: premium
    usage_bucket: gpu-saturated
    resources:
      monetary_cost: negligible
      quota_scarcity: high
      latency: high
      local_compute_occupancy: high
      context_cost: high

  gateway-standard:
    runtime: openai_compatible
    provider: remote_gateway
    cost_class: standard
    usage_bucket: batch-inference
    resources:
      monetary_cost: moderate
      quota_scarcity: moderate
      latency: moderate
      local_compute_occupancy: low
      context_cost: high

cost_rework_policies:
  strict-review:
    max_premium_tasks: 2
    max_premium_retries_per_task: 1
    max_escalations_per_workflow: 1
    allow_higher_cost_reexecution: false
    require_escalation_reason: true

  allow-escalation:
    max_premium_tasks: 3
    max_premium_retries_per_task: 2
    max_escalations_per_workflow: 2
    allow_higher_cost_reexecution: true
    require_escalation_reason: true
```

Machine-readable schema: [`config/providers.schema.json`](../../config/providers.schema.json).

## 5. Public request and result semantics

### 5.1 Request fields (create / delegate)

| Field | Required when | Semantics |
|---|---|---|
| `model_profile` | Optional | Selects operator profile; unknown id rejected at create. |
| `cost_rework_policy` | Optional | Opts into rework governance; unknown id rejected at create. |
| `rework_prior_task_id` | With rework | Prior task in same workflow. |
| `rework_scope` | With rework | `targeted` or `full`. |
| `escalation_reason` | Policy-dependent | Non-empty string when `require_escalation_reason` (default true). |

Rework fields must appear together (`rework_prior_task_id` + `rework_scope`).
Omitting `cost_rework_policy` leaves rework metadata invalid at create.

Independent dimensions remain independently settable, e.g. a `continues` child
with `agent_profile: architecture-reviewer`, `model_profile: local-heavy-premium`,
and `cost_rework_policy: strict-review`.

### 5.2 Projections (privacy-safe)

**`model_profile_resource`** (when present):

- `resolution`, `model_profile`, `cost_class`, `usage_bucket`, `resources` tiers
- Never: credentials, config paths, token counts, pricing secrets

**`cost_policy_status`** (when policy opted in):

- `policy_id`, `preflight_status`, `limits`, `usage`, `remaining`
- `rework`: `prior_task_id`, `scope`, `escalation_reason_present`
- `model_profile_cost_class`, `escalation_decision`
- Never: full `escalation_reason` text on concise/completion surfaces, task prompts, raw artifacts

### 5.3 Rejection reasons (start preflight)

Stable `reason` strings for orchestrator handling:

| `reason` | Meaning |
|---|---|
| `unknown_model_profile_under_policy` | Policy selected but profile unconfigured/unknown |
| `incompatible_model_profile` | Profile runtime/provider/model pin mismatch |
| `missing_escalation_reason` | Required reason absent |
| `unknown_rework_prior_task` | Prior task id not found |
| `cross_workflow_rework_reference` | Prior task belongs to different `root_task_id` |
| `higher_cost_duplicate_reexecution_denied` | Full rework of satisfied prior at higher `cost_class` |
| `premium_task_budget_exceeded` | Workflow premium cap |
| `premium_retry_budget_exceeded` | Per-prior premium retry cap |
| `escalation_budget_exceeded` | Full rework cap |

## 6. Security and privacy

| Concern | Stance |
|---|---|
| **No credential echo** | Cost/profile snapshots never include API keys, env values, or provider config bodies ([RFC-001](RFC-001.md) §7). |
| **No pricing oracle** | Broker does not fetch live rates; operators declare tiers. |
| **Task isolation** | Projections are scoped to the task's artifact directory and MCP responses for that `task_id`. |
| **Reason text** | `escalation_reason` stored in `request.json` for audit; concise paths expose presence only. |
| **No raw artifact injection** | Prior transcripts/files are not auto-forwarded; parents use explicit new tasks with rework metadata. |
| **Deterministic preflight** | Rejections are reproducible from persisted config + workflow history — no model-in-the-loop policy. |

## 7. Compatibility and migration

| Scenario | Behavior |
|---|---|
| **Legacy tasks without `model_profile`** | `model_profile_resource.resolution: unconfigured`, `cost_class: unknown`; unchanged lifecycle. |
| **Legacy tasks without `cost_rework_policy`** | `cost_policy_status.resolution: unconfigured`; no rework caps applied. |
| **Adding policies to existing broker home** | New tasks may opt in; historical tasks without snapshots do not count toward usage until started under policy. |
| **RFC-002 consumers** | `capability_warning` / `capability_contract` unchanged; additive `model_profile_resource` and `cost_policy_status`. |
| **Config validation** | `recollect-lines config validate` and schema cover `model_profiles` / `cost_rework_policies` alongside providers. |

Migration notes for runtime/profile rename remain in
[migration-runtime-profile.md](../migration-runtime-profile.md); `model_profile`
is a distinct operator concept from deprecated legacy `profile` runtime alias.

## 8. Acceptance scenarios

Mapped to [`tests/test_model_profile.py`](../../tests/test_model_profile.py) and
[`tests/test_cost_rework_policy.py`](../../tests/test_cost_rework_policy.py).

| # | Scenario | Expected |
|---|---|---|
| A1 | Omit `model_profile` | `cost_class: unknown`; legacy start succeeds without policy. |
| A2 | Unknown `model_profile` at create | Rejected; no task queued. |
| A3 | Profile runtime/provider/model mismatch at start | `REJECTED` before launch; no launch record. |
| A4 | Premium child, unconfigured parent | Child `cost_class: premium`; parent stays `unknown`. |
| A5 | Omit `cost_rework_policy` | Unconfigured cost policy; legacy behavior. |
| A6 | Policy + unconfigured profile | `REJECTED` at start (`unknown_model_profile_under_policy`). |
| A7 | Targeted premium rework within caps | Start accepted; snapshot shows `scope: targeted`. |
| A8 | Missing `rework_scope` / partial rework fields | Rejected at create. |
| A9 | Cross-root `rework_prior_task_id` | `REJECTED` (`cross_workflow_rework_reference`). |
| A10 | Full rework, satisfied prior, higher cost, default policy | `REJECTED` (`higher_cost_duplicate_reexecution_denied`). |
| A11 | Same as A10 with `allow_higher_cost_reexecution: true` | Start accepted when reason present. |
| A12 | Premium task count at limit | Next premium task `REJECTED`. |
| A13 | Premium retry count at limit | Next premium rework `REJECTED`. |
| A14 | Escalation count at limit | Next `full` rework `REJECTED`. |
| A15 | Concise/completion projections | `escalation_reason_present` without reason text leakage. |
| A16 | Normalized envelope | `broker_observed.model_profile_resource` and `cost_policy_status` additive. |

## 9. Rejected alternatives

| Alternative | Why rejected |
|---|---|
| **Provider billing API integration** | Non-deterministic preflight, credential sprawl, vendor lock-in; contradicts local-first operator control. |
| **Model-name heuristic tables** | Fragile across gateways, aliases, and local routes; encodes folklore as policy. |
| **Infer premium from `agent_profile` or graph depth** | Violates independent-dimension contract; hidden spend. |
| **Broker-autonomous retries on failure/collect** | Duplicates work, breaks parent-directed graphs ([PRD](PRD.md) §3.1). |
| **Semantic duplicate prompt detection** | Unreliable without broker-owned NLP; policy uses declared rework metadata only. |
| **Auto-spawn review tasks from `review-findings` schema** | Conflates output shape with workflow topology; parents retain graph authority. |
| **Inject prior raw artifacts into delegate prompts** | Escalation-only; bypasses caps and secret hygiene. |
| **Post-launch budget enforcement only** | Quota already spent; preflight rejection is required. |
| **Unlimited council/debate loops** | Bounded patterns remain parent-directed ([RFC-001](RFC-001.md) §10.5, [bounded-debate-workflow.md](../bounded-debate-workflow.md)). |

## 10. Operational metrics (future 20–50 task scale test)

When exercising a mixed workflow (investigation → review → targeted fix → optional
full escalation) at **20–50 tasks** under one `external_root_id`, operators
should record:

| Metric | Source | Question it answers |
|---|---|---|
| `premium_tasks` / `remaining.premium_tasks` | `cost_policy_status` snapshots | Did premium cap match intent? |
| `premium_retries_for_prior` | Per-prior counters | Were continuations bounded? |
| `escalations` / `remaining.escalations` | Workflow aggregation | How many full re-runs occurred? |
| `preflight_status` + `reason` on rejections | Terminal `REJECTED` events | Were denials deterministic before launch? |
| `cost_class` distribution | `model_profile_resource` per task | Did spend tiers match operator labels? |
| `local_compute_occupancy` high + `monetary_cost` negligible count | Resource snapshots | Were "free" locals still treated as expensive? |
| Cross-dimension mismatches | Manual audit | Any case where graph role implied cost (should be zero)? |
| Capability vs cost rejections | Separate preflight reasons | Were gates independent? |
| Time-to-reject at start | Event timestamps | Preflight stayed cheap vs adapter runtime? |
| Workflow tree width/depth | `task_tree` / lineage | Lineage limits (`max_children_per_parent`, etc.) still sufficient? |

Suggested pass criteria for the scale test: **zero** adapter launches on
policy-intended rejections; **100%** of premium spend tasks carry configured
`model_profile`; **no** full higher-cost re-executions of satisfied priors unless
explicitly allowed and reasoned.

## 11. Open and deferred items

| Topic | Owner / notes |
|---|---|
| Live token-dollar accounting | Out of scope; operators declare tiers. |
| Automatic model routing from `cost_class` | Deferred; discovery/routing stays parent-directed ([RFC-001](RFC-001.md) §10.5). |
| Semantic rework equivalence | Not planned; caller-declared `rework_scope` only. |
| Per-`usage_bucket` quotas separate from premium caps | Future operator policy extension. |
| Capability-missing → rework escalation coupling | RFC-002 visibility vs RFC-003 spend; parent decides follow-up task. |
| Denial-pattern routing scores | [RFC-002](RFC-002.md) §9 — not inferred into cost policy. |
| Envelope version bump for governance fields | Additive broker_observed keys; no breaking change required for PRs #55/#56. |
| Multi-tenant billing isolation | Product out of scope ([PRD](PRD.md) §8). |

## 12. Explicit non-goals (RFC-003)

- Application code changes in this RFC document PR (implementation landed in #55, #56).
- Provider billing lookup or automatic price calculation.
- Model-name or provider folklore heuristics.
- Autonomous recursion, retries, routing, or debate loops.
- Raw artifact injection into delegate context.
- Automatic review task creation from reviewer-shaped schemas.
- Unrelated documentation cleanup outside RFC-003 and index linkage.

## 13. Relationship to PRD and prior RFCs

- **PRD §3.1 / §5.6** — Parent-directed graphs with deterministic bounds; RFC-003
  adds **resource honesty** and **rework caps** without prescribing topology.
- **RFC-001 §10.5** — Bounded council patterns remain explicit and capped; cost
  policy does not auto-instantiate councils.
- **RFC-002** — Completion integrity and capability contracts are independent
  dimensions from cost class and rework policy; both may apply to the same task.

Implementation evidence: PR #55 (`model_profile.py`, operator docs), PR #56
(`cost_rework_policy.py`, preflight integration, tests). User-facing operator
guides remain the day-to-day reference; this RFC is the durable decision record.
