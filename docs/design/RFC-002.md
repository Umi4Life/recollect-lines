# RFC-002 — Completion Integrity and Capability-Warning Visibility

Status: **accepted decision RFC** (design contract for implementation PRs).
This document records the product and implementation decisions for completion
integrity — making runtime tool-access denials visible without weakening
`read_only` workspace safety. It is **not** the implementation PR.

Related: [PRD.md](PRD.md) (evidence-first collection, §5.3), [RFC-001.md](RFC-001.md)
(trust zones, normalized results, adapter boundary). Resource/cost/rework
governance is explicitly **out of scope** here and belongs in a later
**RFC-003**.

## 1. Problem statement

A delegated runtime may reach a **successful terminal lifecycle state** and
return a fluent natural-language summary that *claims* external verification
(for example, "I checked the remote repository and confirmed the fix is merged")
while one or more tool invocations that would be required to support that claim
were **denied** by the runtime's permission layer.

Today:

- `read_only` / plan-mode enforcement is **correct** for local workspace
  mutation: the broker maps `execution_mode="read_only"` to runtime permission
  profiles that structurally refuse file writes in the supervised workspace
  (see [RFC-001](RFC-001.md) §4.1 and [phase-6a.md](../history/phases/phase-6a.md)).
- The gap is **visibility**, not enforcement: structured denial metadata exists
  in some runtime outputs (for example Claude Code's `permission_denials` in
  the JSON result object) but is buried under `session_metadata` in
  `normalized_result.json`, omitted from `concise_normalized_view`, and absent
  from completion-event summaries. A parent orchestrator polling compact results
  can miss denials entirely and treat a fluent runtime summary as complete
  evidence.

**Completion integrity** means: when structured adapter metadata records tool
access denials, the broker surfaces a concise, privacy-safe capability warning
alongside — not inside — the runtime's prose summary, without conflating
runtime-reported claims with broker-verified evidence.

## 2. Terminology

| Term | Meaning |
|---|---|
| **Completion integrity** | The property that a collected result makes capability limitations visible when structured runtime metadata records them, so success prose cannot be the only signal a parent sees. |
| **Capability observation** | A broker-normalized record that a specific tool identifier was denied during execution. Observations are **broker-reported from adapter metadata**, not broker-verified facts about the external world. |
| **Denied tool identifier** | The runtime's exact tool name string (for example an MCP tool id such as `mcp__provider__tool_name`). PR 1 carries identifiers only — no semantic capability mapping. |
| **Runtime-reported evidence** | Claims in the delegate's summary or structured output, labeled as such — never promoted to broker-verified (PRD §11, RFC-001 §5). |
| **Broker-verified evidence** | Outcomes the broker observed by running its own argv-only commands (`Broker.verify()`). Distinct from capability observations. |
| **`read_only` execution mode** | Local workspace read-only investigation: no direct mutation of the caller's source checkout. **Does not** grant arbitrary MCP or network access; it narrows local file-write tools only. |
| **Required capabilities** | A future contract where a task declares capabilities it needs before launch; enforcement and preflight statuses are **deferred** (§9). |
| **Concise normalized view** | The compact projection of `normalized_result.json` returned by `status`/`collect` and embedded in completion events (`concise_normalized_view()`). |

## 3. Current behavior and observed failure mode

### 3.1 What the broker does today

1. **Adapter collect** — `ClaudeCodeAdapter.collect()` copies
   `permission_denials` from the runtime JSON result into the collected payload
   when present ([phase-6a.md](../history/phases/phase-6a.md)).
2. **Normalization** — `build_normalized_envelope()` stores that array under
   `runtime_reported.session_metadata.permission_denials` when non-null
   (`result_normalization.py`).
3. **Concise view** — `concise_normalized_view()` exposes summary, parser
   warnings, and verification presence — **not** capability observations.
4. **Completion events** — `build_completion_event()` embeds the concise view
   in `result_summary`; denial metadata does not surface.
5. **Lifecycle** — Terminal state follows existing rules (`succeeded`,
   `succeeded_with_warnings`, etc.). Parser/verification gates may downgrade
   success; **permission denials alone do not change state today** (and PR 1
   will not add automatic downgrade — §6.2).

### 3.2 Generic stress-test failure mode (illustrative)

A parent delegates a read-only investigation: *"Verify whether pull request #42
was merged on the upstream default branch."* The runtime is configured with
local read-only workspace access but **no** remote repository MCP tools.

The runtime attempts a denied MCP tool (for example
`mcp__example_host__get_pull_request`). The permission layer records a denial.
The process exits `0`; the JSON result includes a non-empty
`permission_denials` array and a fluent `result` prose summary that states the
PR was merged — inferred from partial local context, not from the denied call.

The parent polls `completion_events` and reads only `result_summary.summary`.
No capability warning appears. The orchestrator treats the task as cleanly
succeeded with a trustworthy-sounding answer, even though a tool required for
that claim never ran.

This is a **visibility and evidence-classification** failure, not a workspace
safety failure: the broker correctly refused local writes; it did not surface
that remote-evidence tools were denied.

### 3.3 Safe workaround today (unchanged)

When remote evidence is required, the **parent** should prefetch it (or delegate
a separate bounded task with an adapter/runtime that actually has the needed
tools) and pass explicit facts in the task instruction. The broker does not
auto-inject raw artifacts or forward credentials on the parent's behalf (PRD
§7, RFC-001 §7). RFC-002 makes denial visibility better; it does not replace
parent-side evidence gathering.

## 4. Proposed design

### 4.1 Data flow (PR 1)

```text
Runtime JSON result
  └─ permission_denials[]          (adapter-specific structured field)
        │
        ▼
Adapter collect payload          (unchanged field name at adapter boundary)
        │
        ▼
Capability observation normalizer  (new; adapter-neutral)
  └─ capability_observations[]   (normalized schema; §4.2)
        │
        ├─► normalized_result.json
        │     runtime_reported.capability_observations
        │     parser.warnings (on partial parse)
        │
        ├─► concise_normalized_view
        │     capability_warning (compact; §4.3)
        │
        └─► completion_events.result_summary
              capability_warning (same compact shape)
```

**PR 1 adapter scope:** `ClaudeCodeAdapter` first. The normalizer schema is
adapter-neutral so later adapters can supply observations without broker
changes.

**Input contract:** Only **structured adapter metadata** is in scope. The
broker must **not** parse/scrape raw prose, stderr, or event logs to infer
denials in PR 1.

### 4.2 Normalized capability observation schema

Each observation is a plain object:

```json
{
  "tool_identifier": "mcp__example_host__get_pull_request",
  "source": "runtime_permission_denial",
  "adapter": "claude_code"
}
```

Rules:

- `tool_identifier` — exact string from structured denial metadata (required,
  non-empty).
- `source` — stable enum string identifying which adapter field produced the
  observation (PR 1: `runtime_permission_denial` from `permission_denials`).
- `adapter` — runtime adapter id (`claude_code`, etc.) for audit only.

**Aggregation** (deterministic):

| Field | Rule |
|---|---|
| `denial_attempt_count` | Total structured denial records processed (including duplicates). |
| `distinct_denied_tool_count` | Count of unique `tool_identifier` values. |
| `denied_tool_identifiers` | Unique identifiers, sorted lexicographically, **at most 16** shown. |
| `truncated` | `true` when `distinct_denied_tool_count > 16`. |
| `denial_attempt_count` | Always the full total; never capped. |

Malformed entries (non-object, missing identifier, wrong types) are skipped;
valid observations are preserved; a parser warning is appended (§4.4).

### 4.3 Compact `capability_warning` object

Surfaced in `concise_normalized_view` and completion-event `result_summary`
when `denial_attempt_count > 0`:

```json
{
  "denial_attempt_count": 3,
  "distinct_denied_tool_count": 2,
  "denied_tool_identifiers": [
    "mcp__example_host__get_pull_request",
    "mcp__example_host__list_commits"
  ],
  "truncated": false
}
```

**Privacy / security boundary for compact surfaces:**

| Allowed | Forbidden |
|---|---|
| Exact denied MCP **tool identifier** strings | Tool **arguments**, file paths, repository names, queries, URLs with secrets |
| Counts (`denial_attempt_count`, `distinct_denied_tool_count`) | Raw stdout/stderr, response bodies, source snippets |
| Boolean `truncated` | Credential material, env values, session tokens |

Full normalized observations may appear in `normalized_result.json` (task-scoped
artifact); compact event/MCP surfaces use `capability_warning` only.

### 4.4 Malformed denial metadata — fail soft

If `permission_denials` (or a future adapter field) is not a list, or contains
unparseable elements:

- Do **not** crash collection or fail the task solely for denial parse errors.
- Emit **zero or more** valid observations from well-formed elements.
- Append a parser warning, for example:
  `permission_denials contained N malformed entries; partial capability observations preserved`.
- `parse_status` follows existing normalization rules; denial parse issues do
  not alone force `failed`.

### 4.5 Envelope version bump

`normalized_result.json` currently uses `envelope_version: 1`
(`result_normalization.ENVELOPE_VERSION`). PR 1 **bumps to `2`** when
`capability_observations` / `capability_warning` are added — additive fields
only; existing consumers can ignore new keys.

Example envelope excerpt after PR 1:

```json
{
  "envelope_version": 2,
  "task_id": "tsk_01H…",
  "state": "succeeded",
  "runtime_reported": {
    "summary": "Pull request #42 appears merged based on local refs.",
    "capability_observations": [
      {
        "tool_identifier": "mcp__example_host__get_pull_request",
        "source": "runtime_permission_denial",
        "adapter": "claude_code"
      }
    ],
    "session_metadata": {
      "permission_denials": ["…raw adapter shape retained for audit…"]
    }
  },
  "parser": {
    "requested_schema": "evidence-report",
    "parse_status": "ok",
    "warnings": []
  },
  "broker_observed": { "…": "…" }
}
```

Concise view excerpt:

```json
{
  "envelope_version": 2,
  "state": "succeeded",
  "summary": "Pull request #42 appears merged based on local refs.",
  "warnings": [],
  "capability_warning": {
    "denial_attempt_count": 1,
    "distinct_denied_tool_count": 1,
    "denied_tool_identifiers": ["mcp__example_host__get_pull_request"],
    "truncated": false
  }
}
```

### 4.6 Lifecycle state — no automatic downgrade (PR 1)

**Decision:** Observed capability denials **do not** automatically change
terminal state from `succeeded` to `succeeded_with_warnings` in PR 1.

Rationale:

- Denials may be expected (for example `read_only` tasks probing unavailable
  remote tools) and parents may still want the runtime summary for partial
  local findings.
- Capability warnings are additive signals; conflating them with parser or
  verification downgrade semantics would mix unrelated concerns.
- A future RFC may define when parents **require** capabilities and how that
  intersects lifecycle (§9).

Existing `succeeded_with_warnings` paths (parser partial/fallback, advisory
verification failure) remain unchanged.

### 4.7 `read_only` scope — do not broaden

RFC-002 and PR 1 **must not** widen what `read_only` means. It remains **local
workspace read-only investigation** under broker policy — not a grant of
arbitrary MCP, network, or repository-mutation access. Surfacing denials makes
missing access visible; it does not authorize new access or credential
forwarding.

## 5. Compatibility, security, and privacy

| Concern | PR 1 stance |
|---|---|
| **Additive API** | Existing result fields and artifact names remain. New keys only. |
| **Envelope version** | Bump `1 → 2`; document in migration notes when implementation lands. |
| **Secret leakage** | Compact surfaces never include denial arguments or tool payloads. |
| **Cross-task isolation** | Observations stay inside the task's artifact directory and MCP responses for that `task_id`. |
| **Trust labeling** | Capability warnings are **runtime-reported observations**, not broker-verified external facts. |
| **Credential forwarding** | **Out of scope** — no generic pass-through of parent tokens to runtimes. |

## 6. Phased implementation plan

### 6.1 PR 1 — Claude Code adapter + normalized visibility (this RFC's first implementation)

Deliverables:

1. Adapter-neutral normalizer from structured `permission_denials` →
   `capability_observations` + `capability_warning`.
2. `ClaudeCodeAdapter` wired as the first producer (field already collected).
3. `build_normalized_envelope()` populates new fields; `ENVELOPE_VERSION = 2`.
4. `concise_normalized_view()` includes `capability_warning` when applicable.
5. `build_completion_event()` passes capability warning through `result_summary`.
6. Deterministic unit tests: aggregation, sort, cap-16/truncated, malformed
   fail-soft, privacy boundary on compact JSON.

Explicit PR 1 non-deliverables:

- Semantic tool → capability ID mapping.
- `required_capabilities` delegate field or preflight enforcement.
- Parsing denials from unstructured logs or prose.
- Automatic `succeeded` → `succeeded_with_warnings` on denials.
- Codex/Cursor/OpenCode adapters (unless they already expose equivalent
  structured fields — otherwise follow-on PRs).
- Generic credential forwarding or MCP proxy expansion.

### 6.2 Later phases (boundaries only — not PR 1)

| Phase | Topic | Notes |
|---|---|---|
| PR 2+ | Additional adapters | Map each runtime's structured denial metadata through the same normalizer. |
| RFC-003+ | Resource/cost/rework governance | Retries, budgets, escalation — **not** RFC-002. |
| Future | Semantic capability IDs | Stable ids such as `repository.remote.read`, backed by per-adapter tool maps. |
| Future | Tool-access profile vs workspace mode | Separate `execution_mode` (workspace mutation) from an explicit tool-access profile. |
| Future | Required-capability contract | Preflight/runtime statuses **distinct** from execution lifecycle states. |
| Future | Evidence provenance lattice | Labels: `orchestrator_supplied`, `runtime_reported`, `broker_observed`, `broker_verified`, `unresolved`. |
| Future | Lifecycle integration | Optional downgrade or `failed` when required capabilities were denied — only after required-capability design lands. |

## 7. Acceptance criteria / test matrix

| # | Scenario | Expected |
|---|---|---|
| A1 | `permission_denials` empty or absent | No `capability_warning`; no observations. |
| A2 | One denial | `denial_attempt_count=1`, one identifier in compact view. |
| A3 | Duplicate denials same tool | `denial_attempt_count` counts all; `distinct_denied_tool_count=1`. |
| A4 | Multiple distinct tools | Sorted identifiers; counts match. |
| A5 | >16 distinct tools | First 16 sorted ids; `truncated=true`; total attempt count preserved. |
| A6 | Malformed list or entries | Valid observations kept; parser warning; collect succeeds. |
| A7 | Terminal `succeeded` with denials | State stays `succeeded`; warning present in concise view. |
| A8 | Completion event payload | `result_summary.capability_warning` matches concise view; no forbidden fields. |
| A9 | `read_only` task with remote-tool denials | Warnings surfaced; workspace still unchanged; no broadened access. |
| A10 | Envelope version | `envelope_version: 2` when new fields present. |
| A11 | Regression | Existing normalization tests pass; additive fields do not break MCP acceptance harness. |

## 8. Alternatives considered and rejected

| Alternative | Why rejected for PR 1 |
|---|---|
| **Scrape denials from prose/logs** | Unreliable, non-deterministic, privacy-risky; violates structured-metadata-only scope. |
| **Auto-fail or auto-`succeeded_with_warnings` on any denial** | Conflates expected denials with errors; breaks parents using partial local results. |
| **Map tools to semantic capabilities immediately** | Premature without per-adapter mapping tables and required-capability contract. |
| **Embed denial details inside `summary`** | Pollutes runtime prose trust zone; breaks concise polling; risks leaking arguments. |
| **Broker-verified capability claims** | Broker does not observe MCP permission decisions; observations stay runtime-reported. |
| **Broaden `read_only` to include MCP allowlists** | Safety regression; tool-access profile is a separate future concern. |
| **Forward parent credentials so denials "go away"** | Out of product scope; increases secret exposure surface. |

## 9. Unresolved / deferred questions

Assigned to **RFC-003 or later** (not RFC-002 / PR 1):

| Question | Owner |
|---|---|
| Cost accounting, retry policy, and rework escalation when capabilities are missing | RFC-003 |
| When should denied required capabilities block launch vs warn at collect? | Future required-capability RFC |
| Canonical semantic capability taxonomy and versioning | Future required-capability RFC |
| Should persistent denial patterns feed adapter routing / discovery scoring? | Phase 6D+ / RFC-003 |
| Per-parent policy: treat capability warnings as hard failures | Host/orchestrator policy, not broker default in PR 1 |
| Normalized provenance field on every evidence item | Future evidence-lattice work |
| Whether `succeeded_with_warnings` should optionally include capability denials | After required-capability contract exists |

## 10. Explicit non-goals (RFC-002)

The following remain **out of scope** for this RFC and its PR 1 implementation:

- Autonomous recursive workflows or broker-run debate loops.
- Unrestricted MCP forwarding or broker-as-proxy for arbitrary tools.
- Broad network access grants beyond what each runtime adapter already enforces.
- Repository mutation in `read_only` tasks.
- Semantic fact-checking of all runtime prose.
- Automatic injection of raw artifacts into the parent context.
- Generic credential forwarding from parent to runtime.
- Lifecycle state-machine rewrite.
- Cost, retry, and escalation policy (RFC-003).

## 11. Relationship to PRD and RFC-001

- **PRD §3.3 / §6** — Collect must distinguish runtime claims from independent
  checks. RFC-002 adds visibility for *missing tool access* without claiming
  broker verification of external facts.
- **RFC-001 §5** — Broker-verified evidence remains argv-only and separate.
  Capability observations are a third, labeled category in the normalized
  envelope (runtime-reported permission events).
- **RFC-001 §7** — Compact warning payloads preserve the existing secret-safe,
  task-scoped stance.

When implementation PRs land, update [RFC-001](RFC-001.md) §8 capability matrix
and add a phase record under [history/phases/](../history/phases/) with test
evidence — that documentation update may accompany the implementation PR, not
this decision RFC.
