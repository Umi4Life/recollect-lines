# RFC-001 — Local Side-Agent Delegation Broker

Status: describes the current (Phase 1–6A) implementation. This is an
implementation RFC, not the product contract — see [PRD.md](PRD.md) for the
provider-neutral requirements this implementation exists to satisfy. Where
this document names a specific stack choice (OpenCode, Python, SQLite, MCP
over stdio), that choice is provisional and revisitable; it is documented
here as "what the code does today and why," not as a permanent requirement.

Source references: [phase-1.md](phase-1.md), [phase-2.md](phase-2.md),
[phase-3.md](phase-3.md), [phase-4.md](phase-4.md), [phase-5b.md](phase-5b.md),
[phase-5c.md](phase-5c.md), [phase-6a.md](phase-6a.md) contain the detailed,
per-phase acceptance criteria and test evidence this RFC summarizes and
consolidates. Where this document and a phase doc disagree on a factual
claim about the code, the phase doc plus the actual source is authoritative;
file an issue.

## 1. Architecture boundary

Three layers, deliberately kept separate:

- **Parent** (out of scope of this repository) decides *what* work to
  delegate and *when* — it calls an interface (MCP tool or CLI command) and
  later asks for status/results. It never talks to storage or subprocesses
  directly.
- **Broker** (`service.py`, `store.py`, `models.py`, `workspace.py`) owns
  task lifecycle mechanics: state machine validation, durable persistence,
  policy enforcement (timeouts, concurrency, allowed modes), workspace
  isolation, and broker-run verification. The broker is runtime-neutral — it
  knows nothing about how to actually run a coding agent.
- **Adapter** (`adapters.py`, `opencode_adapter.py`, `claude_code_adapter.py`, `codex_adapter.py`)
  speaks one specific runtime's language: how to invoke it as a subprocess,
  how to parse its output into a summary, how to signal it to stop.
  `MockAdapter`, `OpenCodeAdapter`, and (as of Phase 6A) `ClaudeCodeAdapter`
  are interchangeable implementations of this boundary; the broker
  (`service.py`) dispatches to whichever adapter a task's `profile` selects
  via a generic `subprocess_adapters` lookup, never a hard-coded
  OpenCode- or Claude-specific branch — see [phase-6a.md](phase-6a.md).

This boundary is what keeps the product contract (PRD.md) independent of any
one runtime: swapping or adding an adapter should never require changing
broker policy or persistence.

## 2. Current chosen stack, and why it is provisional

| Choice | Why it was picked for Phase 1–4 | Why it is not a product requirement |
|---|---|---|
| Python 3.11+ | Fast to iterate, adequate stdlib (`subprocess`, `sqlite3`, `argparse`) for a local-only broker | The broker/adapter boundary is a protocol, not a language; any language with process control and durable storage could implement it |
| SQLite (WAL mode) | Durable, zero-ops, transactional, supports the partial-unique-index trick workspace leases rely on | Any durable store with equivalent transactional guarantees would satisfy the PRD's durability requirement |
| OpenCode CLI (`opencode-ai`) via `npx` | Available, scriptable, produces parseable JSON event output | One adapter is not evidence the architecture only works for one runtime; see §8 capability matrix and known limitations for how far this evidence actually extends |
| MCP over local stdio | Matches how parent agents already expect to discover/call tools, no network surface needed | The PRD requires *an* interface for parent agents, not specifically MCP; a network transport or a different protocol is a future decision, not implied here |

## 3. Task lifecycle, persistence, artifacts

Task state is a validated graph (`models.TaskState`,
`models.ALLOWED_TRANSITIONS`):

```
created → queued → preparing → running → collecting → succeeded
                              ↘        ↘            ↘ succeeded_with_warnings
   ↘ rejected      ↘ cancelled  ↘ cancelling → cancelled
                    (from queued)          ↘ failed
                                  ↘ failed / timed_out (from running)
```

This is the happy-path skeleton, not the full graph: every non-terminal
state also has direct failure/rejection edges (e.g. `queued → rejected`,
`preparing → failed`, `preparing → cancelled`, `collecting → failed`) —
`models.ALLOWED_TRANSITIONS` is the authoritative source for the complete
set. Phase 5B adds one non-terminal state, `recovery_required` (reachable
from `running` and `cancelling`, resolving only to `failed` or `cancelled`),
for a task whose durable launch record can't be confirmed dead after a
broker restart; see [phase-5b.md](phase-5b.md).

Every transition is written as an append-only event alongside the task's
SQLite row (`store.py`); the CLI/MCP `status` call returns the record, the
full event history, and the artifact manifest together. Task artifacts
(`request.json`, `result.json`, `events.jsonl`, `stderr.log`,
`workspace_status.json`, `diff.patch`, `verification.json`) live under a
task-specific directory with a manifest tracking filename, byte count, and
sha256 per file — durable evidence, not just an in-memory result object.
Reconstructing a `Broker`/`TaskStore` against the same home directory
reproduces identical task state (Phase 1 acceptance criterion 5; re-verified
under concurrent/restart scenarios in Phase 3).

## 4. Process-group cancellation and OpenCode compatibility evidence

`OpenCodeAdapter.start()` launches the runtime with `start_new_session=True`
so the child becomes its own process-group leader; the broker stores
`{pid, pgid, command, ...}` as the `task.running` event's metadata.
Cancellation sends `SIGTERM` to the whole group, polls `os.killpg(pgid, 0)`
for a bounded window, and escalates to `SIGKILL` if the group is still
alive — the resulting `CANCELLED` vs. `FAILED` transition is decided by that
*observed* liveness probe, never by the adapter's own exit code or output.

This was exercised against the real `npx --yes opencode-ai@1.17.18` CLI (not
just the local test fixture), including a real multi-process tree
(`npx` → `sh -c` → `opencode`) reaching confirmed `group_terminated: true`.
That same real-CLI exercise is also what surfaced two bugs no fixture-only
unit test caught: the real CLI's nested `part.text` summary shape, and a
liveness-check race from `SIGKILL` being asynchronous (see
[phase-2.md](phase-2.md#real-cli-smoke-verification) for the full account).
This evidence is real but narrow: it covers one adapter, one CLI version
pin, and manual/scripted smoke runs, not continuous compatibility testing
against upstream OpenCode releases.

**Wrapper PID alone is unsafe** — this is a design decision, not an
oversight. A naive implementation might track only the top-level `npx`
PID and consider the task "stopped" once that process exits; this codebase
deliberately tracks and signals the whole process group instead, because
`npx` commonly wraps the real binary through an additional shell layer that
can outlive its parent. Cancellation and cleanup both route through group
liveness checks (`os.killpg(pgid, 0)`) rather than trusting any single PID's
exit status.

### 4.1 Claude Code CLI compatibility evidence (Phase 6A)

`ClaudeCodeAdapter.start()` reuses `OpenCodeAdapter`'s process-group pattern
exactly (`start_new_session=True`, durable `{pid, pgid, command, ...}`
launch record, and the same `cancel_process_group()` implementation — not a
reimplementation) and the same `os.killpg`-observed, never-adapter-reported
cancellation contract.

A real, bounded compatibility spike (three calls) against the installed
`claude` CLI (`2.1.201`) validated the command contract before it was
cemented: `--output-format json` prints exactly one JSON result object on
exit (no incremental stream); `--permission-mode plan` structurally refuses
file writes, confirmed independently via the filesystem, not just the CLI's
own claim; `acceptEdits` is the narrowest mode confirmed to actually write.
A fourth real call exercised the actual adapter/broker end to end against a
second disposable fixture, with the resulting workspace's `HEAD`, porcelain
status, and full tracked-file-tree SHA-256 independently confirmed
unchanged before/after. See [phase-6a.md](phase-6a.md) for the full raw
evidence, command shapes, and honestly-scoped caveats (this covers one CLI
version, a handful of real calls, and no continuous re-verification against
future `claude` releases — the same evidence standard §4 above already
applies to OpenCode).

## 5. Workspace leases, worktrees, and diff/verification evidence

For `execution_mode="isolated_worktree"`, the broker resolves the caller's
workspace to its Git toplevel, captures `base_sha`, and — only after
acquiring a durable lease — creates a broker-owned `git worktree` under
`<home>/worktrees/<task_id>` on a task-specific branch
(`workspace.py`, `store.py`). The adapter is always given the worktree path,
never the original workspace, as its effective working directory.

Leases live in a `workspace_leases` SQLite table with a **partial unique
index** on `canonical_source` filtered to `status = 'active'` — the
concurrency guarantee ("one active writer per source") is enforced by the
database itself, not an in-process lock, so it holds across independent
`Broker` instances. Read-only tasks never acquire a lease and can run freely
alongside an active writer.

Before releasing a worktree/lease on every terminal path, the broker stages
and diffs the worktree against `base_sha` and persists `workspace_status.json`
and `diff.patch` — including the explicit `"clean"` case when nothing
changed. `WorkspaceManager.release()` refuses to remove any path outside its
own worktrees root, so the broker can never be made to delete the source
workspace itself. Cleanup is deliberately skipped — leaving the lease active
(blocking new writers) — whenever an OpenCode process's group can't be
confirmed dead: after a broker restart (`Broker.reconcile()`, Phase 5B,
transitions the task to `recovery_required` instead), or when a cancellation
attempt's group termination wasn't confirmed. See [phase-3.md](phase-3.md)
for the original enumerated test evidence (`tests/test_workspace_safety.py`)
and [phase-5b.md](phase-5b.md) for the restart-reconciliation semantics that
now govern when that lease is finally released, including
byte-identical-source verification and lease persistence across fresh
`Broker` instances.

`Broker.verify(task_id, commands)` runs broker-supplied argv-array commands
(never a shell string) directly via `subprocess.run(..., cwd=working_dir)`
and persists raw stdout/stderr/exit-code evidence as `verification.json`,
marked `broker_verified: true` — because the broker itself executed and
observed it, independent of anything the adapter/runtime claims. As of
Phase 5C, a task's per-task `verification_policy` can also fold this same
`verify()` call's outcome into the task's terminal state instead of
leaving it purely advisory-by-omission — see [phase-5c.md](phase-5c.md).

## 6. MCP stdio tool surface and unsupported live steering

`mcp_server.py` exposes seven tools over newline-delimited JSON-RPC 2.0 on
stdio (`delegate`, `delegate_batch`, `status`, `collect`, `cancel`,
`message`, `reconcile`), each a thin wrapper around the existing `Broker` API
— no lifecycle, policy, or workspace logic is duplicated at this layer. All
protocol output is JSON-RPC on stdout only; diagnostics go to stderr, so a
client reading strict JSON-RPC framing never sees a stray line. See
[phase-4.md](phase-4.md) for the full JSON-RPC error model, the versioned
tool-result envelope, and the tool-by-tool contract.

`message` is the explicit, factual admission that **live task steering is
not supported**: it always returns a structured
`{"status": "unsupported", "reason": "..."}` response rather than silently
dropping the call or pretending OpenCode or Claude Code can accept mid-run
input — neither can, and no adapter here queues, buffers, or forwards message content
anywhere.

## 7. Security / environment / secret handling stance

- No command the broker itself runs (`verify()`) is ever invoked with
  `shell=True`; all such commands are validated as non-empty argv-string
  lists before any of them run.
- The MCP server reads, stores, and echoes no credentials, tokens, or
  environment variables.
- `status`/`collect` responses expose only the requesting task's own
  artifact manifest (filenames, sha256, byte counts) and event metadata —
  never another task's data or a filesystem path outside the broker's
  `--home` directory and the caller's own declared workspace.
- Transport is local stdio only; no network listener is opened by any phase
  implemented so far, and none is currently planned within this RFC's scope.
- Cancellation/liveness probing (`os.killpg`) and worktree operations are
  POSIX/Git-CLI only; there is no Windows-specific process-group or
  libgit2/dulwich fallback implemented.

## 9. Alternatives considered and decisions deferred to future phases

- **Unifying `MockAdapter`/`OpenCodeAdapter` behind one method-shape
  interface** was considered and rejected for Phase 1–4: mock execution is
  instantaneous/synchronous, OpenCode execution is a supervised subprocess;
  forcing a single shape would add indirection without a second real adapter
  to justify the abstraction (see [phase-2.md](phase-2.md#adapter-boundary)).
  Revisit once/if a second real runtime adapter is added.
- **Durable process-handle re-attachment** (surviving a broker restart
  mid-task) was explicitly deferred at the end of Phase 2 and again at the
  end of Phase 3. Phase 5B closes the gap this created (no durable identity
  to reconcile against, `collect()` not idempotent) without ever claiming
  actual reattachment: a new OS process still cannot regain a `Popen`/child
  relationship with an orphaned subprocess, so a still-alive process after a
  restart is reported `recovery_required`, not silently resumed. See
  [phase-5b.md](phase-5b.md).
- **A mandatory verification gate in the lifecycle graph** (forcing a
  verify pass before a task can reach a successful terminal state) was
  explicitly named as future work at the end of Phase 3, and implemented
  in Phase 5C as an opt-in, per-task `verification_policy` — see
  [phase-5c.md](phase-5c.md). It remains opt-in by design, not a global
  default; nothing in this codebase forces every task through `required`.
- **A network/HTTP MCP transport, multi-tenant/auth support** were
  explicitly named out of scope at the end of Phase 4 and remain so; no
  decision to build them has been made.
- **A second heterogeneous runtime adapter** (Claude Code CLI and Codex
  CLI are the original product PRD's preferred candidates) was unscheduled
  as of Phase 5C. A post-Phase-5C roadmap decision sequenced this gap into
  Phase 6A (Claude Code CLI), Phase 6B (Codex CLI), and Phase 6B.5
  (Cursor CLI) — see §10. **Phase 6A is now implemented**
  ([phase-6a.md](phase-6a.md)): `ClaudeCodeAdapter` is the second
  heterogeneous adapter, meeting the PRD's "at least two" MVP boundary
  (PRD.md §9/§10), though all adapters remain marked experimental.
  **Phase 6B is now implemented** ([phase-6b.md](phase-6b.md)):
  `CodexAdapter` is the third adapter. **Phase 6B.5 is now implemented**
  ([phase-6b5.md](phase-6b5.md)): `CursorAdapter` is the fourth adapter.

See [PHASE-5.md](PHASE-5.md) for how 5A–5C (and 6A) were sequenced, and
[phase-5c.md](phase-5c.md)/[phase-6a.md](phase-6a.md) for the detailed
per-phase work delivered. Phase 6B added Codex; Phase 6B.5 added Cursor.
Phase 6C and 6D remain open per §10.

## 10. Roadmap: Phase 6 adapter and provider expansion (6A/6B/6B.5 implemented, 6C/6D not implemented)

This section records a post-Phase-5C roadmap decision. Phase 6A
([phase-6a.md](phase-6a.md)), Phase 6B ([phase-6b.md](phase-6b.md)), and
Phase 6B.5 ([phase-6b5.md](phase-6b5.md)) have been implemented; 6C and 6D
below remain design intent and constraints for future work only.

### 10.1 Two distinct extensibility layers

The roadmap decision draws a hard line between two concepts this codebase
has so far only needed one of:

```text
Runtime adapter
= speaks to and supervises a concrete agent runtime/CLI
  (§1's existing Adapter boundary: process invocation, output parsing,
  cancellation semantics, workspace/permission mapping).

Provider configuration
= describes a model endpoint: base URL, credentials reference, model
  aliases, and declared API capabilities. Not itself a supervised
  coding-agent runtime.
```

A provider configuration entry does not by itself make a safe coding-agent
runtime. A direct OpenAI-compatible API runtime must declare its own
tool/workspace/structured-output capabilities and must never impersonate
Claude Code, Codex, Cursor, or OpenCode. Cursor CLI is a real runtime
adapter (Phase 6B.5); DeepSeek, Qwen, and local OpenAI-compatible endpoints
are named provider-configuration entries (Phase 6C), not adapters.

### 10.2 Revised Phase 6 sequence

| Phase | Scope | Status |
|---|---|---|
| 6A | Claude Code CLI adapter | **Implemented** — see [phase-6a.md](phase-6a.md) |
| 6B | Codex CLI adapter | **Implemented** — see [phase-6b.md](phase-6b.md) |
| 6B.5 | Cursor CLI adapter | **Implemented** — see [phase-6b5.md](phase-6b5.md) |

### 10.3 Phase 6B.5 — Cursor CLI adapter

A real runtime adapter per the §1 boundary, not an OpenAI-compatible-provider
alias. Acceptance evidence will need to cover: authenticated/non-interactive
invocation; structured or safely parsed event/output capture; workspace
semantics and permission mapping; a dedicated process group/session and
cancellation (the same group-liveness discipline §4 already applies to
OpenCode); runtime availability/authentication failure normalization;
model/profile selection where Cursor actually supports it; an artifact and
normalized-result contract; real bounded smoke plus deterministic fixtures
(the same fixture-plus-real-CLI-smoke pattern §4 used for OpenCode). No
claim that a Cursor wrapper/billing label is the actual execution runtime
unless independently verified.

### 10.4 Phase 6C — OpenAI-compatible provider fabric

Plural, named, configuration-driven provider entries (illustrative, not
secret-bearing):

```yaml
providers:
  deepseek:
    kind: openai-compatible
    base_url: https://provider.example/v1
    api_key_env: DEEPSEEK_API_KEY
    default_model: deepseek-reasoner

  qwen:
    kind: openai-compatible
    base_url: https://provider.example/v1
    api_key_env: QWEN_API_KEY
    default_model: qwen-coder

  local:
    kind: openai-compatible
    base_url: http://127.0.0.1:8000/v1
    api_key_env: LOCAL_MODEL_API_KEY
    default_model: local-coder
```

Design constraints carried into whichever future PR implements this phase:

- providers are named and plural, configured, never a hard-coded vendor
  switch statement;
- secrets are referenced by environment/keyring reference only — never
  stored in source, SQLite, task artifacts, summaries, or logs — matching
  §7's existing "no credential echo" stance;
- endpoint/base-URL validation, model aliases, optional safe custom
  headers, timeouts, retry/error normalization, and redaction of any
  secret material from error output;
- explicit, per-provider/per-runtime capability declarations rather than
  an assumption that any chat-completions-shaped endpoint can edit code,
  access a repository, call tools, stream events, or support
  cancellation — a plain OpenAI-compatible endpoint grants none of that by
  default;
- local endpoint trust/TLS policy is documented, not silently weakened
  (e.g. a `http://127.0.0.1` local route is an explicit, documented
  exception, not an implicit default extended to remote endpoints);
- provider selection (which endpoint/model) is a separate concern from
  runtime/adapter selection (which supervised process shape);
- availability/auth failures are structured and provider-specific but
  normalized for the parent, matching the truthful-evidence discipline
  §5/§7 already require of adapters;
- deterministic fake OpenAI-compatible server tests plus bounded
  integration smoke where credentials happen to be available, following
  the §4 real-CLI-smoke precedent.

The initial generic direct-API runtime is deliberately capability-limited
(e.g. structured analysis/review tasks only) until a separately designed
tool-loop/sandbox contract exists; the isolated-worktree writable-workspace
path (§5) remains the domain of adapters that can honestly support it.

### 10.5 Phase 6D — Discovery, routing, and bounded council patterns

Exposes runtime and provider availability/capabilities to hosts, then
supports user/parent-directed patterns already anticipated at the product
level (PRD.md §3.1's "bounded model council comparison"): two independent
plans from different adapters/providers, critique of one result by another
worker, evidence comparison with human/parent synthesis, and lower-cost/
local analysis run alongside higher-capability implementation — always
with an explicit bounded round count, concurrency limit, and cost/time
budget, never an unlimited debate loop. The broker does not pick a
"winner" by personality or vendor; the parent/human chooses the task graph
and acceptance criteria, and the broker records evidence and enforces
policy, consistent with how `Broker.verify()` already separates
broker-observed evidence from runtime self-report (§5, §7).
