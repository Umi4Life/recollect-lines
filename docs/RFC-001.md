# RFC-001 — Local Side-Agent Delegation Broker

Status: describes the current (Phase 1–4) implementation. This is an
implementation RFC, not the product contract — see [PRD.md](PRD.md) for the
provider-neutral requirements this implementation exists to satisfy. Where
this document names a specific stack choice (OpenCode, Python, SQLite, MCP
over stdio), that choice is provisional and revisitable; it is documented
here as "what the code does today and why," not as a permanent requirement.

Source references: [phase-1.md](phase-1.md), [phase-2.md](phase-2.md),
[phase-3.md](phase-3.md), [phase-4.md](phase-4.md) contain the detailed,
per-phase acceptance criteria and test evidence this RFC summarizes and
consolidates. Where this document and a phase doc disagree on a factual
claim about the code, the phase doc plus the actual source is authoritative;
file an issue.

## 1. Architecture boundary

Three layers, deliberately kept separate:

- **Parent** (out of scope of this repository) decides *what* work to
  delegate and *when* — it calls an interface (MCP tool or CLI command) and
  later asks for status/results. It never talks to storage or subprocesses
  directly.
- **Broker** (`service.py`, `store.py`, `models.py`, `workspace.py`) owns
  task lifecycle mechanics: state machine validation, durable persistence,
  policy enforcement (timeouts, concurrency, allowed modes), workspace
  isolation, and broker-run verification. The broker is runtime-neutral — it
  knows nothing about how to actually run a coding agent.
- **Adapter** (`adapters.py`, `opencode_adapter.py`) speaks one specific
  runtime's language: how to invoke it as a subprocess, how to parse its
  output into a summary, how to signal it to stop. `MockAdapter` and
  `OpenCodeAdapter` are two interchangeable implementations of this
  boundary; the broker only ever calls the adapter interface, never anything
  OpenCode-specific.

This boundary is what keeps the product contract (PRD.md) independent of any
one runtime: swapping or adding an adapter should never require changing
broker policy or persistence.

## 2. Current chosen stack, and why it is provisional

| Choice | Why it was picked for Phase 1–4 | Why it is not a product requirement |
|---|---|---|
| Python 3.11+ | Fast to iterate, adequate stdlib (`subprocess`, `sqlite3`, `argparse`) for a local-only broker | The broker/adapter boundary is a protocol, not a language; any language with process control and durable storage could implement it |
| SQLite (WAL mode) | Durable, zero-ops, transactional, supports the partial-unique-index trick workspace leases rely on | Any durable store with equivalent transactional guarantees would satisfy the PRD's durability requirement |
| OpenCode CLI (`opencode-ai`) via `npx` | Available, scriptable, produces parseable JSON event output | One adapter is not evidence the architecture only works for one runtime; see §8 capability matrix and known limitations for how far this evidence actually extends |
| MCP over local stdio | Matches how parent agents already expect to discover/call tools, no network surface needed | The PRD requires *an* interface for parent agents, not specifically MCP; a network transport or a different protocol is a future decision, not implied here |

## 3. Task lifecycle, persistence, artifacts

Task state is a validated graph (`models.TaskState`,
`models.ALLOWED_TRANSITIONS`):

```
created → queued → preparing → running → collecting → succeeded
                              ↘        ↘            ↘ succeeded_with_warnings
   ↘ rejected      ↘ cancelled  ↘ cancelling → cancelled
                    (from queued)          ↘ failed
                                  ↘ failed / timed_out (from running)
```

This is the happy-path skeleton, not the full graph: every non-terminal
state also has direct failure/rejection edges (e.g. `queued → rejected`,
`preparing → failed`, `preparing → cancelled`, `collecting → failed`) —
`models.ALLOWED_TRANSITIONS` is the authoritative source for the complete
set.

Every transition is written as an append-only event alongside the task's
SQLite row (`store.py`); the CLI/MCP `status` call returns the record, the
full event history, and the artifact manifest together. Task artifacts
(`request.json`, `result.json`, `events.jsonl`, `stderr.log`,
`workspace_status.json`, `diff.patch`, `verification.json`) live under a
task-specific directory with a manifest tracking filename, byte count, and
sha256 per file — durable evidence, not just an in-memory result object.
Reconstructing a `Broker`/`TaskStore` against the same home directory
reproduces identical task state (Phase 1 acceptance criterion 5; re-verified
under concurrent/restart scenarios in Phase 3).

## 4. Process-group cancellation and OpenCode compatibility evidence

`OpenCodeAdapter.start()` launches the runtime with `start_new_session=True`
so the child becomes its own process-group leader; the broker stores
`{pid, pgid, command, ...}` as the `task.running` event's metadata.
Cancellation sends `SIGTERM` to the whole group, polls `os.killpg(pgid, 0)`
for a bounded window, and escalates to `SIGKILL` if the group is still
alive — the resulting `CANCELLED` vs. `FAILED` transition is decided by that
*observed* liveness probe, never by the adapter's own exit code or output.

This was exercised against the real `npx --yes opencode-ai@1.17.18` CLI (not
just the local test fixture), including a real multi-process tree
(`npx` → `sh -c` → `opencode`) reaching confirmed `group_terminated: true`.
That same real-CLI exercise is also what surfaced two bugs no fixture-only
unit test caught: the real CLI's nested `part.text` summary shape, and a
liveness-check race from `SIGKILL` being asynchronous (see
[phase-2.md](phase-2.md#real-cli-smoke-verification) for the full account).
This evidence is real but narrow: it covers one adapter, one CLI version
pin, and manual/scripted smoke runs, not continuous compatibility testing
against upstream OpenCode releases.

**Wrapper PID alone is unsafe** — this is a design decision, not an
oversight. A naive implementation might track only the top-level `npx`
PID and consider the task "stopped" once that process exits; this codebase
deliberately tracks and signals the whole process group instead, because
`npx` commonly wraps the real binary through an additional shell layer that
can outlive its parent. Cancellation and cleanup both route through group
liveness checks (`os.killpg(pgid, 0)`) rather than trusting any single PID's
exit status.

## 5. Workspace leases, worktrees, and diff/verification evidence

For `execution_mode="isolated_worktree"`, the broker resolves the caller's
workspace to its Git toplevel, captures `base_sha`, and — only after
acquiring a durable lease — creates a broker-owned `git worktree` under
`<home>/worktrees/<task_id>` on a task-specific branch
(`workspace.py`, `store.py`). The adapter is always given the worktree path,
never the original workspace, as its effective working directory.

Leases live in a `workspace_leases` SQLite table with a **partial unique
index** on `canonical_source` filtered to `status = 'active'` — the
concurrency guarantee ("one active writer per source") is enforced by the
database itself, not an in-process lock, so it holds across independent
`Broker` instances. Read-only tasks never acquire a lease and can run freely
alongside an active writer.

Before releasing a worktree/lease on every terminal path, the broker stages
and diffs the worktree against `base_sha` and persists `workspace_status.json`
and `diff.patch` — including the explicit `"clean"` case when nothing
changed. `WorkspaceManager.release()` refuses to remove any path outside its
own worktrees root, so the broker can never be made to delete the source
workspace itself. Two paths deliberately skip cleanup — an OpenCode process
whose group can't be confirmed dead after a broker restart, and a
cancellation whose group termination wasn't confirmed — leaving the lease
active (blocking new writers) rather than risking deletion under a possibly
still-running process. See [phase-3.md](phase-3.md) for the full enumerated
test evidence (`tests/test_workspace_safety.py`), including
byte-identical-source verification and lease persistence across fresh
`Broker` instances.

`Broker.verify(task_id, commands)` runs broker-supplied argv-array commands
(never a shell string) directly via `subprocess.run(..., cwd=working_dir)`
and persists raw stdout/stderr/exit-code evidence as `verification.json`,
marked `broker_verified: true` — because the broker itself executed and
observed it, independent of anything the adapter/runtime claims.

## 6. MCP stdio tool surface and unsupported live steering

`mcp_server.py` exposes six tools over newline-delimited JSON-RPC 2.0 on
stdio (`delegate`, `delegate_batch`, `status`, `collect`, `cancel`,
`message`), each a thin wrapper around the existing `Broker` API — no
lifecycle, policy, or workspace logic is duplicated at this layer. All
protocol output is JSON-RPC on stdout only; diagnostics go to stderr, so a
client reading strict JSON-RPC framing never sees a stray line. See
[phase-4.md](phase-4.md) for the full JSON-RPC error model, the versioned
tool-result envelope, and the tool-by-tool contract.

`message` is the explicit, factual admission that **live task steering is
not supported**: it always returns a structured
`{"status": "unsupported", "reason": "..."}` response rather than silently
dropping the call or pretending OpenCode can accept mid-run input — it
cannot, and no adapter here queues, buffers, or forwards message content
anywhere.

## 7. Security / environment / secret handling stance

- No command the broker itself runs (`verify()`) is ever invoked with
  `shell=True`; all such commands are validated as non-empty argv-string
  lists before any of them run.
- The MCP server reads, stores, and echoes no credentials, tokens, or
  environment variables.
- `status`/`collect` responses expose only the requesting task's own
  artifact manifest (filenames, sha256, byte counts) and event metadata —
  never another task's data or a filesystem path outside the broker's
  `--home` directory and the caller's own declared workspace.
- Transport is local stdio only; no network listener is opened by any phase
  implemented so far, and none is currently planned within this RFC's scope.
- Cancellation/liveness probing (`os.killpg`) and worktree operations are
  POSIX/Git-CLI only; there is no Windows-specific process-group or
  libgit2/dulwich fallback implemented.

## 8. Capability / limitation matrix

| Capability | Status | Evidence / caveat |
|---|---|---|
| Durable task state across restarts | Supported | SQLite-backed; Phase 1 criterion 5, re-verified Phase 3 |
| Mock adapter lifecycle | Supported | Deterministic, synchronous, fully unit-tested |
| Real OpenCode adapter lifecycle | Supported, experimental | One real-CLI smoke round per phase-2.md; not continuously re-verified against upstream OpenCode releases |
| Process-group cancellation | Supported (POSIX) | Confirmed via `os.killpg` liveness probe, not adapter self-report |
| Durable re-attachment to a running adapter process after a broker restart | **Not supported** | Process handles are in-memory only (Phase 2 known limitation); a restart loses the ability to `collect()`/`cancel()` an in-flight OpenCode task, though the task row/events persist and cleanup fails safe (worktree/lease left active) rather than deleting live work |
| Workspace isolation for writable tasks | Supported | Partial-unique-index lease + broker-owned worktree; source proven byte-identical before/after in tests |
| Concurrent writers to the same source | Rejected (one active writer per source) | Enforced in SQLite, not an in-process lock |
| Broker-run verification with raw evidence | Supported | Argv-only, `broker_verified: true`, kept distinct from runtime self-report |
| Mandatory verification before success | **Not supported** | Verification is caller-invoked, not a required lifecycle gate (tracked for Phase 5C) |
| MCP stdio interface | Supported | Six tools, versioned envelope, real-subprocess tested |
| Live mid-task steering | **Not supported, by design** | `message` always returns a factual `unsupported` response |
| Windows support | **Not supported** | POSIX process groups and Git-CLI worktree operations only |
| Remote/distributed workers | **Not implemented, not planned in this RFC** | See PRD §8 out-of-scope |

## 9. Alternatives considered and decisions deferred to future phases

- **Unifying `MockAdapter`/`OpenCodeAdapter` behind one method-shape
  interface** was considered and rejected for Phase 1–4: mock execution is
  instantaneous/synchronous, OpenCode execution is a supervised subprocess;
  forcing a single shape would add indirection without a second real adapter
  to justify the abstraction (see [phase-2.md](phase-2.md#adapter-boundary)).
  Revisit once/if a second real runtime adapter is added.
- **Durable process-handle re-attachment** (surviving a broker restart
  mid-task) was explicitly deferred at the end of Phase 2 and again at the
  end of Phase 3; the current mitigation is fail-safe cleanup (never delete
  a possibly-live process's worktree), not actual reattachment. Deferred to
  Phase 5B.
- **A mandatory verification gate in the lifecycle graph** (forcing a
  verify pass before a task can reach a successful terminal state) was
  explicitly named as future work at the end of Phase 3. Deferred to Phase
  5C.
- **A network/HTTP MCP transport, multi-tenant/auth support** were
  explicitly named out of scope at the end of Phase 4 and remain so; no
  decision to build them has been made.

See [PHASE-5.md](PHASE-5.md) for how the two open items above are
sequenced.
